1.2 Descripción General

El Sistema de Gestión de Contenido (CMS) del DIF San Mateo Atenco es una aplicación web institucional desarrollada a medida en PHP 8 con base de datos MySQL 8. Su propósito es permitir que el personal administrativo de la institución gestione de forma autónoma, segura y eficiente todos los contenidos del sitio web público del DIF, sin necesidad de conocimientos técnicos de programación.

El sistema provee un panel de administración protegido por autenticación de sesiones PHP, con protección CSRF, control de intentos de inicio de sesión y un sistema de permisos granular por sección. El sitio público consume los datos almacenados en MySQL y los presenta mediante plantillas PHP con Bootstrap 5, animaciones CSS y componentes JavaScript modernos.

La arquitectura sigue un patrón de diseño similar a MVC (Modelo-Vista-Controlador) implementado de forma procedimental en PHP, donde cada módulo del panel de administración actúa como controlador, las consultas PDO representan el modelo y las plantillas HTML/PHP constituyen la vista.

1.3 Objetivos del Sistema

  • Gestión autónoma de contenidos: Permitir al personal del DIF actualizar noticias, programas, trámites, galería fotográfica, sliders y documentos de transparencia sin intervención técnica externa.
  • Transparencia gubernamental: Publicar y mantener actualizados los documentos de transparencia exigidos por la normativa: SEAC, Cuenta Pública, Presupuesto Anual, PAE, Matrices de Indicadores, CONAC e información financiera.
  • Seguridad institucional: Garantizar que solo personal autorizado acceda al panel de administración, con registro de auditoría completo de todas las acciones realizadas.
  • Accesibilidad ciudadana: Ofrecer al ciudadano información actualizada sobre servicios, trámites, programas sociales, voluntariado y comunicación social del DIF.
  • Escalabilidad modular: Permitir la incorporación de nuevas secciones de transparencia mediante el módulo de secciones dinámicas sin modificar el código fuente.
  • Compatibilidad con hosting compartido: Funcionar correctamente en entornos de hosting cPanel con PHP 8.x y MySQL 8, sin dependencias de servidor adicionales.

1.4 Alcance Funcional

El sistema cubre los siguientes dominios funcionales:

  • Gestión de contenido del sitio público (sliders, programas, trámites, noticias, galería, voluntariado)
  • Administración de información institucional (presidencia, direcciones, organigrama)
  • Módulos de transparencia gubernamental (SEAC, Cuenta Pública, Presupuesto Anual, PAE, Matrices, CONAC, Financiero)
  • Gestión de avisos de privacidad
  • Configuración del footer y contacto del sitio
  • Administración de usuarios y permisos del panel
  • Registro de auditoría y reportes de actividad
  • Subida y gestión de imágenes y documentos PDF
Sección 02

Arquitectura del Sistema

2.1 Visión General de la Arquitectura

El sistema implementa una arquitectura de tres capas (presentación, lógica de negocio y datos) sobre un stack LAMP (Linux/Apache/MySQL/PHP) en producción y WAMP/XAMPP en desarrollo local. La comunicación entre capas es síncrona mediante peticiones HTTP estándar y asíncrona mediante llamadas AJAX para operaciones de actualización parcial de la interfaz.

El patrón arquitectónico adoptado es un MVC procedimental simplificado: cada archivo PHP del directorio admin/ actúa simultáneamente como controlador (procesa el POST, ejecuta la lógica de negocio) y como vista (renderiza el HTML de respuesta). Las consultas a base de datos se encapsulan mediante PDO con sentencias preparadas, actuando como capa de modelo.

2.2 Capa de Presentación (Frontend)

La capa de presentación está compuesta por plantillas PHP que generan HTML5 dinámico. El sitio público utiliza Bootstrap 5 como framework CSS base, complementado con hojas de estilo personalizadas ubicadas en css/. La tipografía principal es Montserrat (Google Fonts), con fuentes de respaldo del sistema.

Los componentes interactivos del frontend incluyen:

  • Swiper.js: Carrusel principal de la página de inicio con transiciones suaves y autoplay configurable.
  • OwlCarousel2: Carrusel secundario para la sección "DIF Comunica" con soporte responsive.
  • Lightbox2: Visualizador de imágenes en modal para la galería fotográfica.
  • WOW.js + Animate.css: Animaciones de entrada al hacer scroll en elementos de la página.
  • PDF.js: Visualizador de documentos PDF embebido en el panel de administración.
  • jQuery: Utilizado para manipulación DOM, AJAX y compatibilidad con plugins. Se carga desde CDN con fallback local en lib/jquery/jquery.min.js.

2.3 Capa de Lógica de Negocio (Backend PHP)

La lógica de negocio reside en los archivos PHP del directorio admin/. Cada módulo sigue el siguiente flujo de procesamiento:

  1. Inclusión de auth_guard.php para verificar autenticación y permisos.
  2. Inclusión de csrf.php para generación y validación de tokens CSRF.
  3. Inclusión de includes/db.php para obtener la conexión PDO singleton.
  4. Procesamiento del método POST: validación de entradas, sanitización y ejecución de la acción solicitada.
  5. Registro automático de la acción en el historial de auditoría (admin_historial).
  6. Renderizado de la vista HTML con los datos actualizados desde la base de datos.

2.4 Capa de Datos (MySQL / PDO)

La capa de datos utiliza MySQL 8.0.45 con el charset utf8mb4_unicode_ci para soporte completo de Unicode, incluyendo emojis y caracteres especiales del español. El acceso a la base de datos se realiza exclusivamente mediante PDO (PHP Data Objects) con las siguientes configuraciones de seguridad:

PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION
PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC
PDO::ATTR_EMULATE_PREPARES   => false  // Sentencias preparadas nativas

La función get_db() en includes/db.php implementa el patrón Singleton para reutilizar la conexión PDO durante toda la ejecución del script, evitando múltiples conexiones innecesarias a la base de datos.

Aviso de Seguridad Esta sección describe los mecanismos de seguridad implementados. La información aquí contenida es de uso interno exclusivo. No debe compartirse con personal no autorizado.

6.1 Autenticación Basada en Sesiones PHP

El sistema utiliza sesiones PHP nativas para gestionar la autenticación de los administradores. El flujo de autenticación es el siguiente:

  1. El usuario envía credenciales (usuario + contraseña) mediante POST con token CSRF.
  2. Se verifica el token CSRF antes de procesar cualquier dato.
  3. Se consulta la tabla admin por el username proporcionado.
  4. La contraseña se verifica con password_verify($password, $hash) contra el hash bcrypt almacenado.
  5. Si la autenticación es exitosa, se regenera el ID de sesión con session_regenerate_id(true) para prevenir session fixation.
  6. Se establecen las variables de sesión: $_SESSION['admin_logged'], $_SESSION['admin_id'], $_SESSION['admin_username'], $_SESSION['admin_rol'].
  7. Se registra el evento de login en admin_historial.
// Verificación de contraseña con bcrypt
if ($admin && password_verify($password, $admin['password'])) {
    session_regenerate_id(true);
    $_SESSION['admin_logged'] = true;
    $_SESSION['admin_id']     = (int) $admin['id'];
    $_SESSION['admin_rol']    = $admin['rol'] ?? 'admin';
    header('Location: dashboard');
    exit;
}

6.2 Protección CSRF (Cross-Site Request Forgery)

Todos los formularios del panel de administración incluyen un token CSRF criptográficamente seguro. La implementación en admin/csrf.php utiliza las siguientes características:

  • Generación: bin2hex(random_bytes(32))  64 caracteres hexadecimales, 256 bits de entropía.
  • Almacenamiento: Array de hasta 10 tokens en $_SESSION['csrf_tokens'] para soportar múltiples pestañas abiertas simultáneamente.
  • Validación: hash_equals($stored, $token)  comparación en tiempo constante para prevenir ataques de timing.
  • Consumo: El token se elimina del array tras su uso (parámetro $consume = true).
  • Rotación: Se genera un nuevo token en cada carga de formulario.
// Generación de token CSRF
function csrf_token(): string {
    $token = bin2hex(random_bytes(32));
    $_SESSION['csrf_tokens'][] = $token;
    if (count($_SESSION['csrf_tokens']) > 10) {
        array_shift($_SESSION['csrf_tokens']); // FIFO: eliminar el más antiguo
    }
    return $token;
}

// Validación con comparación en tiempo constante
function csrf_validate(string $token, bool $consume = true): bool {
    foreach ($_SESSION['csrf_tokens'] as $i => $stored) {
        if (hash_equals($stored, $token)) {
            if ($consume) unset($_SESSION['csrf_tokens'][$i]);
            return true;
        }
    }
    return false;
}

6.3 Protección contra Fuerza Bruta

El sistema implementa rate limiting basado en dirección IP para el formulario de inicio de sesión:

  • Se registra cada intento fallido en la tabla login_attempts con la IP y timestamp.
  • Antes de procesar las credenciales, se cuenta el número de intentos fallidos de la IP en los últimos 15 minutos.
  • Si el contador supera 5 intentos, se bloquea el acceso durante 15 minutos con mensaje de error genérico.
  • Los intentos fallidos también se registran en logs/login_attempts.log para auditoría.
// Rate limiting: máximo 5 intentos por IP en 15 minutos
$stmtRate = $pdo->prepare(
    'SELECT COUNT(*) FROM login_attempts
     WHERE ip = ? AND attempted_at > DATE_SUB(NOW(), INTERVAL 15 MINUTE)'
);
$stmtRate->execute([$ip]);
$attempts = (int) $stmtRate->fetchColumn();
if ($attempts >= 5) {
    $error = 'Demasiados intentos fallidos. Espere 15 minutos.';
}

6.4 Auto-Logout por Inactividad

El sistema implementa cierre de sesión automático por inactividad en dos niveles:

  • Nivel PHP (servidor): auth_guard.php verifica que el tiempo transcurrido desde $_SESSION['last_activity'] no supere 300 segundos (5 minutos). Si se supera, destruye la sesión y redirige a login?expired=1.
  • Nivel JavaScript (cliente): El sidebar incluye un timer JavaScript de 5 minutos que se reinicia con cualquier evento de usuario (mousemove, keydown, click, scroll, touchstart). Al expirar, redirige a login.php?expired=1.

6.5 Sistema de Permisos por Sección

Los usuarios con rol usuario solo pueden acceder a las secciones del panel para las que tienen un registro en admin_permisos. La verificación se realiza en auth_guard.php:

// Verificar permiso para la sección actual
$stmt_guard = $pdo->prepare(
    'SELECT id FROM admin_permisos WHERE user_id = ? AND seccion_file = ?'
);
$stmt_guard->execute([$_SESSION['admin_id'], $current_admin_file_guard]);
if (!$stmt_guard->fetch()) {
    header('Location: dashboard'); // Denegar acceso

}

6.6 Protección contra SQL Injection

El sistema utiliza exclusivamente sentencias preparadas PDO con parámetros enlazados (? o :nombre) para todas las consultas a la base de datos. La opción PDO::ATTR_EMULATE_PREPARES => false garantiza que las sentencias preparadas sean nativas del motor MySQL, no emuladas por PHP, lo que proporciona protección real contra inyección SQL.

// Ejemplo de consulta segura con PDO
$stmt = $pdo->prepare('SELECT id, username, password FROM admin WHERE username = ? LIMIT 1');
$stmt->execute([$username]);
$admin = $stmt->fetch(); // Nunca se concatena $username directamente en el SQL

6.7 Protección contra XSS (Cross-Site Scripting)

El sistema aplica las siguientes medidas contra XSS:

  • Salida HTML: Todos los datos dinámicos se escapan con htmlspecialchars($valor, ENT_QUOTES, 'UTF-8') antes de renderizarse en HTML.
  • Contenido TinyMCE: El contenido HTML generado por el editor se almacena tal cual en la base de datos y se renderiza sin escapar solo en contextos donde se espera HTML (usando echo $contenido directamente). Se confía en que TinyMCE sanitiza el input del editor.
  • Respuestas JSON: Los datos en respuestas AJAX se codifican con json_encode() que escapa caracteres especiales por defecto.

6.8 Gestión Segura de Subida de Archivos

El módulo admin/upload_handler.php implementa un proceso de validación multicapa:

  1. Verificación del código de error PHP (UPLOAD_ERR_OK)
  2. Validación de tamaño máximo (20 MB imágenes, 50 MB PDFs)
  3. Validación de extensión de archivo contra lista blanca
  4. Validación del tipo MIME real con finfo_file() (no confía en el MIME declarado por el cliente)
  5. Generación de nombre de archivo sanitizado (sin caracteres peligrosos)
  6. Uso de move_uploaded_file() (función segura de PHP para archivos subidos)
  7. Compresión y redimensionamiento automático de imágenes con GD Library

6.11 Consideraciones de Seguridad Adicionales

Las siguientes prácticas de seguridad están implementadas o se recomiendan para el entorno de producción:

Protección del archivo config.php

El archivo config.php contiene credenciales de base de datos y debe protegerse de acceso público. El comentario en el archivo indica que debe moverse fuera del webroot en producción. Como medida alternativa, se puede agregar la siguiente regla en .htaccess:

Protección del directorio uploads/

El directorio uploads/ almacena archivos subidos por los administradores. Para prevenir la ejecución de scripts PHP maliciosos que pudieran subirse, se recomienda agregar un .htaccess en ese directorio:

Options -ExecCGI

Enumeración de usuarios

El mensaje de error del formulario de login es genérico ("Usuario o contraseña incorrectos") y no revela si el usuario existe o no en la base de datos. Esto previene la enumeración de usuarios válidos del sistema.

Protección contra clickjacking

La cabecera X-Frame-Options: SAMEORIGIN previene que el sitio sea embebido en un iframe de otro dominio, protegiendo contra ataques de clickjacking donde un atacante superpone el sitio legítimo sobre una página maliciosa.

Validación de entradas en el servidor

Todas las entradas del usuario se validan y sanitizan en el servidor PHP, independientemente de cualquier validación del lado del cliente (JavaScript). Las validaciones del cliente son solo para mejorar la experiencia de usuario, nunca como medida de seguridad.

  • Campos numéricos: (int)$_POST['id']  conversión explícita a entero
  • Campos de texto: trim(htmlspecialchars($_POST['titulo']))  eliminación de espacios y escape HTML
  • Campos de fecha: validación de formato con DateTime::createFromFormat()
  • Slugs: preg_replace('/[^a-z0-9\-]/', '', strtolower($slug))  solo caracteres URL-seguros

Sección 10

Gestión de Archivos

10.1 Proceso de Subida de Imágenes

Toda subida de archivos pasa por la función centralizada handle_upload(array $file, string $type) en admin/upload_handler.php. El proceso completo es:

  1. Verificación de error PHP: Se comprueba que $file['error'] === UPLOAD_ERR_OK.
  2. Validación de tamaño: Imágenes: máximo 20 MB (UPLOAD_MAX_IMAGE_MB). PDFs: máximo 50 MB (UPLOAD_MAX_PDF_MB).
  3. Validación de extensión: Lista blanca: jpg, jpeg, png, webp para imágenes; pdf para documentos.
  4. Validación MIME real: finfo_file($finfo, $file['tmp_name']) detecta el tipo MIME real del archivo, independientemente de la extensión declarada.
  5. Sanitización del nombre: Se eliminan caracteres peligrosos con preg_replace('/[^a-zA-Z0-9_\-\. áéíóúñÁÉÍÓÚÑ]/', '_', $baseName).
  6. Resolución de conflictos: Si ya existe un archivo con el mismo nombre, se agrega un sufijo numérico incremental.
  7. Creación de directorio: Si el directorio de destino no existe, se crea con mkdir($destDir, 0755, true).
  8. Movimiento seguro: move_uploaded_file($file['tmp_name'], $destPath).
  9. Compresión automática: Las imágenes se comprimen y redimensionan con GD Library (máx. 1920px de ancho, JPEG/WEBP al 85% de calidad, PNG compresión nivel 6).

Flujo de Datos Detallado por Módulo

Flujo: Publicación de Noticia

El siguiente flujo describe el proceso completo desde que el administrador sube una noticia hasta que aparece en el sitio público:

  1. El administrador accede a admin/noticias.php. auth_guard.php verifica la sesión activa y los permisos de sección.
  2. Se genera un token CSRF y se renderiza el formulario de nueva noticia con campos: título, descripción, fecha y archivo de imagen.
  3. El administrador completa el formulario y hace clic en "Guardar".
  4. El navegador envía una petición POST multipart a admin/noticias.php con action=add.
  5. auth_guard.php intercepta el POST, valida el token CSRF y registra la acción en admin_historial.
  6. El módulo valida los campos requeridos (título, fecha).
  7. Si se adjuntó imagen, se llama a handle_upload($_FILES['imagen'], 'image'): validación MIME, tamaño, extensión, sanitización del nombre, movimiento a uploads/images/ y compresión GD.
  8. Se ejecuta la sentencia PDO preparada: INSERT INTO noticias_imagenes (titulo, descripcion, fecha_noticia, imagen_path) VALUES (?, ?, ?, ?).
  9. Se establece un mensaje flash de éxito en $_SESSION['flash_message'] y se redirige con header('Location: noticias').
  10. El sitio público comunicacion-social/noticias.php consulta SELECT * FROM noticias_imagenes ORDER BY fecha_noticia DESC y renderiza las noticias con Lightbox2.

Flujo: Subida de Documento de Transparencia (PDF)

  1. El administrador accede al módulo de transparencia correspondiente (ej: admin/seac.php).
  2. Selecciona el bloque anual y el concepto al que desea adjuntar el PDF.
  3. Hace clic en "Subir PDF" y selecciona el archivo desde su equipo.
  4. El formulario envía POST con action=upload_pdf, concepto_id y el archivo PDF.
  5. handle_upload($_FILES['pdf'], 'pdf') valida: error PHP, tamaño máximo 50 MB, extensión .pdf, MIME real application/pdf.
  6. El archivo se mueve a uploads/pdfs/ con nombre sanitizado.
  7. Se inserta el registro en la tabla correspondiente (ej: seac_pdfs) con la ruta relativa del PDF.
  8. En el panel admin, el PDF aparece en la lista con un botón "Ver" que abre el modal visor (iframe con PDF.js).
  9. En el sitio público, el PDF se enlaza con un botón de descarga/visualización.

Flujo: Reordenamiento con SortableJS

  1. El administrador arrastra un elemento de la lista en el panel admin.
  2. SortableJS actualiza el orden visual en el DOM y dispara el evento onEnd.
  3. El handler JavaScript serializa el nuevo orden como array JSON: [{"id":3,"orden":1},{"id":1,"orden":2}].
  4. Se envía una petición AJAX POST con action=reorder, csrf_token y el array JSON.
  5. El servidor valida el CSRF, decodifica el JSON y ejecuta un UPDATE por cada elemento: UPDATE tabla SET orden = ? WHERE id = ?.
  6. Se retorna {"success":true} y el frontend muestra una notificación de éxito.
  7. El sitio público consulta los registros con ORDER BY orden ASC y los presenta en el nuevo orden.

Flujo: Creación de Sección de Transparencia Dinámica

  1. El administrador accede a admin/transparencia_dinamica.php.
  2. Completa el formulario: nombre de la sección, slug URL, plantilla a reutilizar (ej: seac), ícono Bootstrap Icons.
  3. El sistema valida que el slug sea único en trans_secciones.
  4. Se inserta el registro en trans_secciones con la plantilla seleccionada.
  5. La nueva sección aparece automáticamente en el menú de transparencia del sitio público.
  6. Al acceder a la sección, el sistema carga la plantilla correspondiente y consulta los datos de las tablas trans_bloques, trans_conceptos y trans_pdfs filtrados por seccion_id.
  7. El administrador puede gestionar el contenido de la nueva sección desde admin/transparencia_seccion.php?id=[seccion_id].

Flujo: Gestión de Álbum Fotográfico

  1. El administrador crea un nuevo álbum en admin/galeria.php con nombre, fecha y portada.
  2. Se inserta en galeria_albumes y se sube la imagen de portada a uploads/images/.
  3. El administrador accede al álbum y sube imágenes individuales (subida múltiple soportada).
  4. Cada imagen se procesa con handle_upload() y se inserta en galeria_imagenes con referencia al album_id.
  5. El sitio público comunicacion-social/galeria.php consulta los álbumes activos y sus imágenes.
  6. Las imágenes se presentan en una cuadrícula con Lightbox2 para visualización en modal con navegación entre fotos del álbum.

Consideraciones de Integridad de Datos

El sistema implementa las siguientes medidas para mantener la integridad de los datos:

  • Eliminación en cascada: Las tablas con relaciones de clave foránea usan ON DELETE CASCADE para eliminar registros dependientes automáticamente (ej: eliminar un álbum elimina todas sus imágenes en galeria_imagenes).
  • Eliminación de archivos físicos: Al eliminar un registro con archivo asociado, el sistema elimina también el archivo del servidor con unlink().
  • Unicidad de slugs: Los slugs de trámites y secciones de transparencia tienen restricción UNIQUE KEY para evitar duplicados.
  • Transacciones: Las operaciones que involucran múltiples tablas (ej: crear álbum + subir portada) se realizan en secuencia con manejo de errores para revertir cambios parciales.
  • Validación de existencia: Antes de eliminar un archivo físico, se verifica que el archivo exista con file_exists() para evitar errores de PHP.

Características Técnicas Implementadas

  • Conexión PDO singleton con charset utf8mb4 y sentencias preparadas nativas
  • Protección CSRF con tokens criptográficos de 256 bits y soporte multi-pestaña
  • Rate limiting de login: máximo 5 intentos por IP en ventana de 15 minutos
  • Auto-logout por inactividad: 5 minutos (PHP + JavaScript)
  • Compresión automática de imágenes: máx. 1920px, JPEG/WEBP 85%, PNG nivel 6
  • Validación MIME real con PHP finfo (no confía en extensión del cliente)
  • Registro de auditoría automático con detección de dispositivo y navegador
  • Sistema de permisos granular por archivo PHP de sección
  • Sidebar colapsable con grupos y estado activo persistente
  • Reordenamiento drag & drop con SortableJS en todos los módulos aplicables
  • Editor TinyMCE 6 integrado en todos los módulos con contenido HTML enriquecido
  • Soporte de múltiples pestañas del navegador con pool de tokens CSRF
  • Mensajes flash de sesión para retroalimentación tras operaciones POST
  • Output buffering para prevenir errores "headers already sent"
  • Fallback de jQuery local cuando el CDN no está disponible

Glosario Técnico

TérminoDefinición
Apache Servidor web HTTP de código abierto. En este sistema, procesa las peticiones HTTP, aplica las reglas de .htaccess y ejecuta los scripts PHP mediante mod_php o PHP-FPM.
bcrypt Algoritmo de hashing de contraseñas basado en Blowfish. Incorpora un "salt" aleatorio y un factor de costo configurable que hace que el cómputo sea intencionalmente lento, dificultando los ataques de fuerza bruta. PHP lo implementa con password_hash() y password_verify().
Bootstrap 5 Framework CSS de código abierto para desarrollo web responsive. Proporciona un sistema de grid de 12 columnas, componentes UI prediseñados (modales, alertas, botones, formularios) y utilidades CSS.
CDN (Content Delivery Network) Red de servidores distribuidos geográficamente que sirven contenido estático (CSS, JS, fuentes) desde el servidor más cercano al usuario, reduciendo la latencia de carga.
CMS (Content Management System) Sistema de Gestión de Contenido. Software que permite crear, editar, organizar y publicar contenido digital sin necesidad de conocimientos de programación.
CONAC Consejo Nacional de Armonización Contable. Órgano de gobierno que emite las normas y lineamientos contables para los entes públicos en México.
cPanel Panel de control de hosting web que proporciona una interfaz gráfica para gestionar sitios web, bases de datos, correos electrónicos, dominios y configuraciones del servidor.
CSRF (Cross-Site Request Forgery) Ataque que engaña al navegador del usuario para que envíe peticiones no autorizadas a un sitio web en el que está autenticado. Se previene con tokens únicos por formulario.
DSN (Data Source Name) Cadena de conexión que especifica el tipo de base de datos, host, nombre de base de datos y charset para PDO. Formato: mysql:host=localhost;dbname=nombre;charset=utf8mb4.
finfo Extensión PHP que detecta el tipo MIME real de un archivo analizando su contenido binario (magic bytes), independientemente de la extensión del archivo o el tipo MIME declarado por el cliente.
GD Library Biblioteca de PHP para creación y manipulación de imágenes. Permite redimensionar, comprimir y convertir imágenes en formatos JPEG, PNG y WEBP.
hash_equals() Función PHP que compara dos cadenas en tiempo constante, independientemente de su longitud. Previene ataques de timing que podrían revelar información sobre el valor secreto comparado.
htaccess Archivo de configuración de Apache a nivel de directorio. Permite configurar reglas de reescritura de URLs, protección de directorios, cabeceras HTTP y otras directivas sin modificar la configuración global del servidor.
InnoDB Motor de almacenamiento de MySQL que soporta transacciones ACID, claves foráneas con integridad referencial y bloqueo a nivel de fila. Es el motor por defecto en MySQL 8.
jQuery Biblioteca JavaScript que simplifica la manipulación del DOM, el manejo de eventos, las animaciones y las peticiones AJAX. Ampliamente utilizada para compatibilidad con plugins como OwlCarousel2 y Lightbox2.
Lightbox2 Plugin jQuery que muestra imágenes en un modal superpuesto sobre la página, con navegación entre imágenes de una galería. Utilizado en la sección de galería fotográfica del sitio público.
MIME (Multipurpose Internet Mail Extensions) Estándar que define el tipo de contenido de un archivo. Ejemplos: image/jpeg, application/pdf, text/html. En el sistema, se valida el MIME real del archivo subido con finfo.
mod_rewrite Módulo de Apache que permite reescribir URLs dinámicamente mediante expresiones regulares. Utilizado para crear URLs amigables y redirigir peticiones.
MVC (Model-View-Controller) Patrón de arquitectura de software que separa la aplicación en tres componentes: Modelo (datos y lógica de negocio), Vista (presentación) y Controlador (manejo de peticiones). El sistema implementa una variante procedimental simplificada de este patrón.
OwlCarousel2 Plugin jQuery para crear carruseles/sliders responsive con soporte táctil. Utilizado en la sección "DIF Comunica" del sitio público.
PAE (Programa Anual de Evaluación) Documento de planeación que establece los programas y acciones a evaluar durante el ejercicio fiscal, conforme a la normativa de evaluación del desempeño gubernamental.
PDO (PHP Data Objects) Extensión PHP que proporciona una interfaz uniforme para acceder a diferentes motores de base de datos. Soporta sentencias preparadas que previenen inyección SQL.
PDF.js Biblioteca JavaScript de Mozilla para renderizar documentos PDF directamente en el navegador sin plugins externos. Utilizada en el panel de administración para previsualizar PDFs.
Rate Limiting Técnica de seguridad que limita el número de peticiones que un cliente puede realizar en un período de tiempo determinado. En este sistema, limita los intentos de login a 5 por IP en 15 minutos.
Salt Valor aleatorio único que se agrega a la contraseña antes de aplicar el hash. Previene ataques de diccionario y rainbow tables. bcrypt genera e incorpora el salt automáticamente en el hash resultante.
SEAC Sistema de Evaluación y Acreditación de la Calidad. Marco de referencia para evaluar y mejorar la calidad de los servicios públicos municipales en el Estado de México.
Session Fixation Ataque en el que un atacante establece el ID de sesión de un usuario antes de que se autentique, para luego usar ese ID y acceder a la sesión autenticada. Se previene con session_regenerate_id(true) tras el login.
Singleton Patrón de diseño que garantiza que una clase tenga una única instancia durante la ejecución del programa. En este sistema, la conexión PDO es un singleton implementado con una variable estática en get_db().
Slug Versión URL-friendly de un texto, generalmente en minúsculas, sin espacios ni caracteres especiales. Ejemplo: "Cuenta Pública 2024" ? cuenta-publica-2024. Utilizado para identificar trámites y secciones de transparencia en las URLs.
SortableJS Biblioteca JavaScript para implementar funcionalidad de arrastrar y soltar (drag & drop) en listas y grids. Utilizada en el panel de administración para reordenar elementos de sliders, programas, galería y otros módulos.
SQL Injection Ataque que inserta código SQL malicioso en los parámetros de entrada de una aplicación para manipular las consultas a la base de datos. Se previene con sentencias preparadas PDO.
Swiper.js Biblioteca JavaScript moderna para crear carruseles/sliders con soporte táctil, transiciones CSS3 y API flexible. Utilizada en el carrusel principal de la página de inicio.
TinyMCE Editor de texto enriquecido (WYSIWYG) basado en JavaScript. Permite a los usuarios crear y editar contenido HTML sin conocer el lenguaje de marcado. Versión 6 utilizada en este sistema.
Token CSRF Valor aleatorio único generado por el servidor e incluido en cada formulario HTML. Al procesar el formulario, el servidor verifica que el token enviado coincida con el almacenado en la sesión, confirmando que la petición proviene del sitio legítimo.
utf8mb4 Codificación de caracteres MySQL que soporta el conjunto completo de Unicode, incluyendo emojis y caracteres de idiomas asiáticos. Supera las limitaciones de utf8 de MySQL que solo soporta caracteres de 3 bytes.
WEBP Formato de imagen moderno desarrollado por Google que ofrece mejor compresión que JPEG y PNG con calidad visual equivalente. Soportado por todos los navegadores modernos.
WOW.js Biblioteca JavaScript que activa animaciones CSS (de Animate.css) cuando los elementos entran en el viewport al hacer scroll. Utilizada para efectos de entrada en el sitio público.
XAMPP Paquete de software de código abierto que incluye Apache, MySQL, PHP y Perl. Utilizado como entorno de desarrollo local en Windows para este proyecto.
XSS (Cross-Site Scripting) Ataque que inyecta scripts maliciosos en páginas web vistas por otros usuarios. Se previene escapando los datos dinámicos con htmlspecialchars() antes de renderizarlos en HTML.
